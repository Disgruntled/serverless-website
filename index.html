This website has no server

Ludacris, I know but bare with me for a moment. This website is powered entirely using cloudflare workers. Cloudflare workers implements the Javascript ServiceWorkers API server side (or more specifically, at the edge)

It's a serverless computing solution that's a lot of fun

It's intended use is to offload a bit of work from your busy servers. You're saving a lot of extra hops when you use a worker, either in your browser or in cloudflare to fulfill your request

It's particularly attractive where you want to write a single page app, but don't want to put all the trust in your user's user-agent.

This was done by creating cloudflare workers with a default route of https://exit23a.ca/* and http://exit23a.ca/*

The actual IP address that's set to "orange cloud" in cloudflare for my account is a random IP for google.com. The service worker intercepts the request and I've configure it to never send the actual traffic to google

Faq

Do I recommend this?

Definitely not. Not only is it needlessly complicated, it also breaks a lot of cloudflares built in functionality. A great example is that I had to use a "page rule" to upconvert all traffic to HTTPS because the built in cloud flare functionality is broken.

Another example: workers are evaluated before cloud flare apps, but after page rules. So "apps" are also out.

Then why do it?

A good way to get test out my own knowledge of javascript and to just see if I could do something quirky

Why cloudflare? Why not lambda at the edge or lambda + apigateway

Cloudflare is simpler. AWS is incredible, but there is a lot of overhead to making something work.

Where is the data stored

In Cloudflare workers key-value (KV)pairs. Such that the content itself is stored in cloudflare.

How do you update this website

I use the cloudflare workers key value API to overwrite the entry that stores this file. It's quite powerful, with minimal effort this could be made into a dynamic CMS
